# SCIM 2.0 — System for Cross-domain Identity Management

**Canonical source:** IETF RFC 7643 (core schema) and RFC 7644 (protocol)
(`rfc:7643`, `rfc:7644`), September 2015.
**Role in the crosswalk:** the provisioning half of human authentication —
keeping identities and their lifecycle in sync across systems.

## What it defines
SCIM standardizes a REST API and JSON schema for provisioning and
de-provisioning users and groups: `/Users`, `/Groups`, filtering, PATCH
semantics, and a common resource schema. An identity provider pushes
create/update/deactivate to downstream applications through one protocol.

## Why it is a coupling anchor
Entra ID, Okta, Google Cloud Identity and others speak SCIM to provision into
downstream systems, so joiner/mover/leaver flows — the accountability backbone
of "who has access" — port across IdPs and clouds. Deactivation propagating
promptly is a governance control, not a convenience.

## Caveat
SCIM provisions accounts; it does not authenticate sessions (that is OIDC/SAML)
or authorize actions (that is policy). Inconsistent SCIM support across
applications is common, so reconciliation/audit is still required.
